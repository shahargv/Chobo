# Chobo UI Test Data

Use these values for the default single-node UI journey. They are derived from `TestingSuite/Tests/BackupRestoreSingleNode` and `docs/Configuration.md`.

## Environment

- ChoboServer container host: `choboserver`
- Published browser URL: chosen by `start-ui-env.ps1`, usually `http://127.0.0.1:<free-port>`
- Bootstrap environment:
  - `CHOBO_INIT_ADMIN_USER=` empty
  - `CHOBO_INIT_ACCESS_TOKEN=` empty
  - Do not use `dev-static-access-token`
  - Do not use `static-test-token`
- MinIO alias inside Docker network: `backup-s3`
- MinIO endpoint for Chobo: `http://backup-s3:9000`
- MinIO bucket: `data-bucket`
- MinIO access key: `chobo-access-key`
- MinIO secret key: `chobo-secret-key`

## ClickHouse

Source cluster form:

- Name: `ui-source`
- Mode: `Single instance`
- Access nodes: `clickhouse-source:9000`
- Max DOP: `1`
- TLS: off
- Username: empty
- Password: empty

Restore cluster form:

- Name: `ui-restore`
- Mode: `Single instance`
- Access nodes: `clickhouse-restore:9000`
- Max DOP: `1`
- TLS: off
- Username: empty
- Password: empty

Source SQL:

```sql
DROP DATABASE IF EXISTS backup_single_source SYNC;
CREATE DATABASE IF NOT EXISTS backup_single_source ENGINE = Atomic;
CREATE TABLE backup_single_source.source_orders
(
    id UInt32,
    name String
)
ENGINE = MergeTree
ORDER BY id;
INSERT INTO backup_single_source.source_orders (id, name) VALUES
(1, 'alpha'),
(2, 'beta'),
(3, 'gamma');
```

Restore verification SQL:

```sql
SELECT id, name
FROM backup_single_restore.restored_orders
ORDER BY id
FORMAT CSV
```

Expected restored CSV:

```csv
1,"alpha"
2,"beta"
3,"gamma"
```

## Backup Storage

- Name: `ui-minio`
- Endpoint: `http://backup-s3:9000`
- Bucket: `data-bucket`
- Region: `us-east-1`
- Path prefix: `ui/<RunId>` or empty
- Addressing: path style
- Access key: `chobo-access-key`
- Secret key: `chobo-secret-key`

After save, the UI must not show the secret key in plain text.

## Policy

- Name: `ui-orders-policy`
- Source cluster: `ui-source`
- Backup storage: `ui-minio`
- Failed backups: keep
- Selector:
  - Action: Include
  - Database kind: exact
  - Database value: `backup_single_source`
  - Table kind: exact
  - Table value: `source_orders`
- Retention:
  - Full retention minutes: empty
  - Incremental retention minutes: empty
  - Min backups to keep: `1`
  - Min full backups: `1`

The selector preview should include `backup_single_source.source_orders`.

## Schedule

- Name: `ui-daily-full`
- Policy: `ui-orders-policy`
- Backup type: Full
- Timezone: `UTC`
- Preset: Daily
- Hour: `2`
- Minute: `0`
- Enabled: true

Use the UI-generated cron expression and verify the UI says it is valid and shows next runs.

## Restore

- Source backup: the latest successful full backup from `ui-orders-policy`
- Target cluster: `ui-restore`
- Layout: Single node or the default valid single-node option
- Source table: `backup_single_source.source_orders`
- Target database: `backup_single_restore`
- Target table: `restored_orders`
- Mode: Schema + data
- Append: false
- Allow schema mismatch: false

## Optional Sharded Values

Use only for the `sharded` scenario:

- Source access node: `clickhouse-source-s1-r1:9000`
- Restore access node: `clickhouse-restore-s1-r1:9000`
- Source cluster name token: generated by the system-test harness as `{source.ClusterName}`
- Source table: `backup_sharded_source.orders_local`
- Selector: Include exact database `backup_sharded_source`, exact table `orders_local`
